## Tier B (model-judged judgment)

Run on 2026-06-13. **Method:** for each scenario a fresh **solver** subagent (Opus 4.8, no access to
the gold/rubric) was given only the solver-visible bundle (prompt + normalized transcripts + project
code + git log + manifest) and asked to produce `ground-truth.md` under the truth-hierarchy rules.
Each output was then scored against the withheld `judge_only` gold using `judge_prompt.md`
(claim-support: every `must` satisfied, no `must_not` violated, no false-precision/fabricated
citations, code-wins enforced globally).

**Scored twice, independently:**
1. **Orchestrator self-score** — 23 / 23 pass.
2. **Independent judge subagents** — one fresh judge per scenario, each given only the candidate
   output + the gold bundle + `judge_prompt.md`, told to be adversarial and *not* told the expected
   outcome. Each re-derived the evidence itself (e.g. the quantitative-drift judge independently
   `grep`-counted the 23 commands) and returned a structured verdict. **23 / 23 pass.**

**Result: 23 / 23 pass (100%), confirmed by independent judges.**

## Multi-tier run — Haiku / Sonnet / Opus (2026-06-13, decision-history family)

Ran the supersession family (N=5/25/100/250, tools-enabled) on **Haiku** and **Sonnet** solvers (Opus
already covered), to test whether the suite discriminates by model tier.

| N | Haiku | Sonnet | Opus |
|---|-------|--------|------|
| 5   | ✓ (ADR 0002) | ✓ | ✓ |
| 25  | ✓ (ADR 0012) | ✓ | ✓ |
| 100 | ✓ (ADR 0050) | ✓ | ✓ |
| 250 | ✓ (ADR 0125) | ✓ | ✓ |

**Finding: the tools-enabled supersession family does NOT discriminate Haiku→Opus — all three tiers
pass at every size.** Every solver, given `grep`/`Read` and told to search, found the correct buried
needle ADR and ruled out the Postgres analytics-warehouse near-miss. This is partly a *good* product
result (the CLI's `search`/`get` + grep make the task tractable even for a small model) but it means
this family is a **ceiling check, not a discriminator**. Real tier separation needs the **no-tools
long-context variant** (`run_evals.py --emit-bundle` writes `library.txt` for exactly this) and/or
harder traps — see `CALIBRATION.md`. That variant is built but not yet run at scale here.

### A scoring lesson (recorded in ADR 0022)

The first attempt scored these runs with a regex over the free-form `ground-truth.md`. It produced
**wrong verdicts on real outputs** — false negatives ("ADR 0125\n…\nSuperseded by ADR 0251" across a
title) and false positives ("migrated *from* Postgres" read as "Postgres is current"). A heuristic
that can invert the result is worse than none. `score.py` was rebuilt: authoritative scoring now
requires a structured `rosetta-verdict` block the solver emits (checked exactly against the
fixture-derived needle); absent that, it reports `scorable: false` and routes to the LLM judge rather
than guessing. The 8 runs above predate that contract, so they are **read-verified** (manually
inspected), not auto-scored — an authoritative re-run would ask each solver for the verdict block.

### Caveats specific to the drift run (honest)

- **The solver had `grep`/`Bash`**, so "find the needle in 250 ADRs" is a *retrieval-with-tools* task,
  not pure long-context reading. A flat curve here means "the model reliably searches when it has
  search tools and is told to." It does **not** show long-context-recall limits; to test those you'd
  withhold search tools and force the library into the prompt.
- **One run per size point** — this is the measurement apparatus and a ceiling check, not a
  statistical curve. Drift would surface with a weaker solver model, a degraded SKILL prompt, or
  no-tools long-context; the harness is built to catch those. The size axis exists so that when a
  regression *does* bend the curve, it's visible.

### Caveats (honest)

- **Judge independence — addressed.** Both a solver pass and an independent judge pass were run as
  isolated subagents. Solvers never received the gold; judges received the gold but not the expected
  verdict, and re-verified each claim against the evidence themselves. Residual caveat: all agents are
  the same model family (Opus 4.8), so this guards against bias-in-context, not against a systematic
  blind spot shared across instances. A different judge model would harden it further.
- **Single solver model / single run:** one Opus-4.8 pass per scenario; no multi-sample or
  multi-model averaging. A weaker solver model would likely score lower — that's the point of the suite.
- **Tier B is not in CI** (non-deterministic, needs model access); Tier A is the CI gate. Reproduce
  Tier B with `python3 run_evals.py --emit-bundle <dir>` then run a solver + an independent judge per
  bundle.
